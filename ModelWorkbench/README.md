# ModelWorkbench

This directory is the offline training workspace for the MQL5AutoTrader system. It is where
OHLCV data is fetched, labels are designed, features are engineered, neural networks are
trained, and production models are evaluated — all before anything is deployed to the live
engine in `Engine/`.

## Philosophy

The approach is built around a single core question: **when, on a 1-minute bar, does a
stop-entry trade have a genuinely high probability of hitting its take-profit before its
stop-loss?**

Rather than predicting price direction in the abstract, every label and every model is
grounded in a concrete, simulated trade. Labels are assigned by scanning forward through
actual H/L candle data to see whether a stop-entry at the bar's High or Low would have hit
its take-profit first. The model is then asked to learn what market conditions — observable
from past price action and indicators — distinguish those winning entries from the losers.

Several principles flow from this:

**Precision over recall.** A handful of high-confidence trades beats a flood of marginal
ones. The loss function is explicitly designed to reward precision on trade signals, with
only a soft floor on recall to prevent the model ignoring a direction entirely.

**Trend-aware entry, not pure breakout.** Signals are generated by looking for mean-reversion
pullbacks *within* a confirmed trend — buying dips in uptrends, selling rallies in downtrends.
Range-bound markets are skipped. This improves label quality because trades entered with
the macro trend have higher TP-hit rates in back-simulation.

**No lookahead, anywhere.** Every indicator, every multi-timeframe feature, every regime
signal is computed causally from past data only. Higher-timeframe features are explicitly
shifted by one completed HTF bar before being merged back to the base timeframe.

**Label and live inference must match.** The SL/TP distances used at label time are the
same distances used for live order sizing. The scaler fitted on training data is serialised
into the model pack and reloaded at inference. The feature column list is also saved in the
pack. A model is only useful if every part of the pipeline is self-consistent.

## Workflow Overview

```
1. Fetch data          →  1_0 Get Historical Data.ipynb  (or fetch_datasets_bulk.py)
       ↓
2. Design labels       →  1_1 Signal Lab.ipynb
       ↓
3. Select features     →  1_2 Feature Lab.ipynb
       ↓
4. Verify parity       →  1_3 Feature Parity Check.ipynb
       ↓
5a. Train (notebook)   →  2_0_a Train LSTM.ipynb  /  2_0_b Train TCN.ipynb
  or
5b. Train (script)     →  train_prod_model_cli.py  (production — all config from JSON profiles)
                       →  train_sweep_tcn.py / train_sweep_lstm.py  (hyperparameter search)
       ↓
6. Evaluate offline    →  3_0 Review Model - Multiclass.ipynb
       ↓
7. Review live trades  →  4_0 Production Trade Report.ipynb
```

The output of step 5 is a `.pkl` model pack written to `Engine/Model Packs/`. The live
engine loads this pack directly — there is no separate deployment step.

### 1_1 Signal Lab

**Purpose:** Interactively tune the labelling parameters that control *which* bars become
BUY/SELL signals before any model is trained.

Getting labels right is the most important step in the whole pipeline — a model can only be
as good as the targets it is trained on. This notebook lets you visualise the effect of each
parameter on a price chart before committing to a configuration.

**What you can tune here:**

| Parameter group | What it controls |
|---|---|
| **Regime detection** | `ma_period`, `slope_threshold`, `atr_percentile` — how aggressively to filter out sideways/low-volatility markets |
| **Triple-barrier sizing** | `tp_mult`, `sl_mult`, `max_horizon` — the risk-reward ratio and time limit for each simulated trade |
| **Entry selectivity** | `trend_pullback_thresh` — raise to require a deeper pullback before labelling an entry; fewer but higher-quality signals |

**How labels work:**

Each bar is first assigned a market regime (uptrend / downtrend / range) using a causal EMA
slope with ATR gating. In production, range bars are skipped entirely. Within trending bars,
signals are generated for pullback entries — buying dips in uptrends, selling rallies in
downtrends — based on a rolling Z-score of price.

Once a candidate signal is identified, the labeller scans forward through subsequent candle
High/Low data to see which barrier is hit first:
- **TP hit first** → label = winning trade (BUY or SELL depending on direction)
- **SL or time limit hit first** → label = FLAT (no trade)

The resulting label distribution is shown in the chart. Aim for a TP-hit rate where winning
BUY and SELL signals each represent roughly 10–20% of total bars. Much less suggests
overly restrictive parameters; much more suggests labels are too easy.

Finalized parameters are saved to `params/label_params.json` and loaded by the training
notebooks automatically.

### 1_2 Feature Lab

**Purpose:** Assess which features carry genuine predictive signal using conventional ML
models (Random Forest, LightGBM, Logistic Regression) before investing time in deep
learning training runs.

This notebook answers: *do these features actually separate BUY/SELL from FLAT?* If
conventional models struggle, a neural network will too — and you should revisit the labels
or feature engineering first.

**What you can do here:**

- Load any OHLCV dataset and apply the same labelling config from `1_1 Signal Lab`
- Compare multiple conventional classifiers on the multiclass task
- Inspect feature importances and classification reports per class
- Identify features that add noise or are redundant

This is also a good place to prototype new feature ideas cheaply before adding them to
`Learn/features.py` and running a full training sweep.

### 1_3 Feature Parity Check

**Purpose:** Verify that features computed in a single bulk pass (as done during training)
are numerically identical to features computed on a rolling sliding window (as done during
live inference).

This is a critical correctness test. Multi-timeframe resampling, rolling statistics, and
certain causal indicators can produce subtly different values depending on how much history
is available at computation time. If a feature diverges between the bulk and sliding paths,
it will perform differently live than in backtest — a silent source of model degradation.

**How it works:**

1. Computes features on the full dataset in one bulk pass (training path)
2. For a sample of bars, computes features on a rolling window of fixed length (live path)
3. Aligns both on timestamp and measures the max absolute difference per feature column
4. Any feature with non-negligible divergence is flagged for review or removal

Run this notebook whenever a new feature is added to `Learn/features.py`, especially MTF
or rolling-window features.

### 2_0_a Train LSTM – Multiclass

**Purpose:** Training template for the `LSTMAttentionSEClassifier` — a bidirectional LSTM
with Squeeze-Excite recalibration and multi-head attention pooling.

**Architecture summary:**

```
BiLSTM (256 hidden, 3 layers)
  → Squeeze-Excite channel recalibration (context window = 32 bars)
  → Multi-head attention pooling (8 heads, learned query)
  → Dropout → Linear(512, 3)
```

The notebook walks through the full training pipeline in order:

| Section | What happens |
|---|---|
| Configuration | Dataset path, model hyperparameters, label params, sequence length, loss weights |
| Load dataset | Read CSV, sort by time |
| Generate labels | `causal_triple_barrier_hilow_trend_labeler` assigns BUY / FLAT / SELL |
| Calculate outcomes | Per-bar TP/SL outcome vectors used by the loss function |
| Feature engineering | `add_all_features()` — full ~100-feature set including MTF indicators |
| Train/val split | Fixed temporal tail held out as validation (never shuffled) |
| Preprocessing | `RobustScaler` on continuous features; one-hot flags passed through unscaled |
| Training loop | AdamW optimiser, focal + precision + recall-floor + direction-confusion loss |
| Evaluation | Per-class precision/recall/F1, confusion matrix, simulated P&L on validation set |
| Save model pack | Serialises weights, scaler, feature list, and metadata to `Engine/Model Packs/` |

**Use this notebook** when experimenting with LSTM architecture changes or for a new symbol
where you want interactive visibility into each training step.

### 2_0_b Train TCN – Multiclass

**Purpose:** Same training pipeline as `2_0_a`, but using the `TCNAttentionSEClassifier` —
a dilated causal Temporal Convolutional Network.

**Architecture summary:**

```
Dilated causal TCN stack (256 channels, 6 layers, kernel=3)
  [dilations: 1, 2, 4, 8, 16, 32 → receptive field ≈ 127 bars (~2 hours at M1)]
  [each layer: causal conv → weight norm → GELU → dropout → residual add]
  → Squeeze-Excite channel recalibration
  → Multi-head attention pooling (8 heads, learned query)
  → Dropout → Linear(256, 3)
```

The TCN is **fully causal** by design — left-padding only, no future context within the
stack. It also trains significantly faster than the LSTM on long sequences because it
parallelises across the time dimension.

**Receptive field:** at `kernel=3, layers=6` the model sees ~127 bars of history. To cover
a full trading session (~480 bars), increase to 8 layers (receptive field ≈ 511 bars).

The notebook structure is identical to `2_0_a`. Choose TCN over LSTM when:
- Training speed matters (e.g., large datasets or many hyperparameter experiments)
- You want guaranteed causal consistency within the sequence window
- Sweep results favour TCN for the target symbol

### 3_0 Review Model – Multiclass

**Purpose:** Offline model evaluation on a held-out dataset, with detailed P&L simulation
and per-class analysis.

Load any saved model pack, point it at a validation CSV, and this notebook will:

- Run the full feature engineering and preprocessing pipeline using the pack's own saved
  scaler and feature list (guaranteeing consistency with training)
- Sweep the full dataset with a sliding inference window (seq_len=256 bars)
- Apply configurable probability thresholds for BUY and SELL signals
- Simulate all trades with customisable commission and the same SL/TP distances used at
  label time
- Produce a trade-by-trade breakdown, cumulative P&L curve, per-class precision/recall,
  and confusion matrix

This is the primary tool for deciding whether a newly trained model is ready for live
deployment. A model should show positive simulated P&L on validation data it has never
seen, with precision on BUY and SELL signals meaningfully above 50%, before going live.

### 4_0 Production Trade Report

**Purpose:** Post-trade analysis of live performance directly from the MetaTrader 5 terminal.

Requires a running MT5 terminal with deal history. The notebook:

- Pulls closed deal history via the MT5 Python API for a configurable date range and
  optional symbol filter
- Produces a per-symbol summary table (net P&L, win rate, trade count, average R)
- Renders an interactive OHLCV candlestick chart with entry/exit markers overlaid
- Shows daily net P&L and a cumulative P&L curve
- Lists best and worst closed trades

Use this notebook regularly to compare live performance against the offline validation
metrics from `3_0 Review Model`. A significant divergence between the two is a signal to
investigate data quality, execution slippage, or model drift.

## Training Scripts

### Production trainer — `train_prod_model_cli.py`

`train_prod_model_cli.py` is the **primary production training script**. All configuration is
driven by JSON profile files under `params/` — no hardcoded values. Each run is fully
reproducible by re-supplying the same CLI flags.

**Supported symbols:** `EURUSD`, `US500`, `XAUUSD`, `US2000`, `NAS100`, `SpotCrude`

**Supported architectures:** `LSTM` (`LSTMAttentionSEClassifier`), `TCN` (`TCNAttentionSEClassifier`)

**Minimal invocation:**

```powershell
# From repo root
.\.venv\Scripts\python.exe .\ModelWorkbench\train_prod_model_cli.py `
    --symbol EURUSD `
    --label-profile EURUSD_1m_dev `
    --model-arch LSTM `
    --model-profile EURUSD_1m_r11 `
    --loss-profile EURUSD_1m_r11 `
    --patience 12 `
    --epochs 30
```

Training config (`seq_len`, `batch_size`, `lr`, `weight_decay`) is read from the model profile
in `params/model_params_multiclass.json`. Any of these can be overridden on the CLI.

**Key CLI flags:**

| Flag | Default | Description |
|---|---|---|
| `--symbol` | required | Instrument — maps to OHLCV CSV and feature function |
| `--label-profile` | required | Key in `params/label_params.json` |
| `--model-arch` | `LSTM` | `LSTM` or `TCN` |
| `--model-profile` | `prod` | Architecture + training config profile in `params/model_params_multiclass.json` |
| `--loss-profile` | `prod` | Loss hyperparameter profile in `params/loss_params_multiclass.json` |
| `--epochs` | `30` | Maximum training epochs |
| `--patience` | `7` | Early-stop patience (epochs without val-loss improvement) |
| `--n-rows` | all | Use only the last N rows (useful for fast test runs) |
| `--seq-len` | from profile | Override sequence length |
| `--batch-size` | from profile | Override batch size |
| `--lr` | from profile | Override learning rate |
| `--weight-decay` | from profile | Override AdamW weight decay |
| `--resume-model-pack` | none | Path to a `.pkl` to resume training from |
| `--no-cloud-log` | false | Disable cloud log mirroring |

Run `python train_prod_model_cli.py --help` for the full argument list.

**Output** (written to `Engine/Model Packs/`):

| File | Contents |
|---|---|
| `*_model.pkl` | Full model pack (best val-loss checkpoint) |
| `*_best_pnl_model.pkl` | Model pack at the epoch with highest val-set PnL (if different) |
| `*_summary.json` | Training metrics, config, and per-epoch curves |
| `*_plots.png` | Val loss, precision/recall, confusion matrix, PnL curves |

Training progress is logged to `Engine/train_multiclass_prod.log` (and optionally mirrored to
`CLOUD_LOG_DIR` from `.env`). Each epoch line shows gradient norm, prediction distribution,
per-direction PnL, and all four loss components.

## The Learn Package

`Learn/` contains all reusable Python modules shared between the notebooks, training scripts,
and (via a mirror copy) the live engine in `Engine/Learn/`.

| Module | Responsibility |
|---|---|
| `features.py` | `add_all_features()` — full ~100-feature set. Per-symbol variants (`_add_features_EURUSD`, etc.) wrap this with symbol-specific `regime_params`. |
| `labels.py` | `causal_triple_barrier_hilow_trend_labeler()` — the primary labeller. `causal_market_regime()` — regime detection. `calculate_trade_outcomes_all_candles()` — Numba-compiled TP/SL outcome scanner. |
| `preprocess.py` | `preprocess_ohlcv()` — RobustScaler routing, one-hot pass-through, PR_ normalisation. |
| `Models.py` | `LSTMAttentionSEClassifier`, `TCNAttentionSEClassifier`, `TransformerClassifier`, `TransformerSEClassifier`, `HybridLSTMTransformer`. |
| `Loss.py` | `TradeProfitabilityLoss` — focal CE + precision penalty + recall floor hinge + direction confusion penalty. |
| `Loaders.py` | `SequenceDataset` — sliding-window PyTorch dataset. |
| `data.py` | `fetch_ohlcv()`, `fetch_ohlcv_bulk()` — cTrader API data download. |
| `Util.py` | Shared training utilities (logging, checkpoint helpers, metrics). |
| `report.py` | Helpers for the Production Trade Report notebook. |

> **Important:** `ModelWorkbench/Learn/` and `Engine/Learn/` are expected to be identical.
> Any change to a module in one tree must be mirrored in the other.

## Model Packs

Trained models are saved as `.pkl` files to `Engine/Model Packs/`. Each pack is a Python
dictionary containing everything needed to run the model identically at inference:

| Key | Contents |
|---|---|
| `model` | PyTorch `state_dict` |
| `model_params` | Constructor kwargs to recreate the architecture |
| `model_info` | Metadata: `model_type`, `seq_len`, symbol, training date, val metrics |
| `scaler` | Fitted `RobustScaler` from the training split |
| `feature_function` | Reference to the feature engineering function |
| `feature_cols` | Ordered list of feature column names (selected at inference) |
| `preprocess_function` | Reference to `preprocess_ohlcv` |
| `preprocess_args` | Kwargs passed to preprocessing at inference |
| `regime_params` | Regime detection parameters (if used as a feature) |

Alongside each pack, a `*_summary.json` is written with training metrics and configuration
for inspection without unpickling. A `*_plots.png` is also saved showing training curves
and the confusion matrix.
