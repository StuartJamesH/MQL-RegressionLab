# Minor Uplift: Incremental Improvements to LGBM Regression Pipeline

## Executive Summary

The current LightGBM regression pipeline achieves holdout Spearman of ~0.30 on quality targets (R² ~0.08) with modest trading-signal performance. This plan identifies 11 specific gaps spanning data leakage, label quality, feature scaling, metric design, and model architecture—each backed by direct reference to the actual code. Proposed fixes are grouped into three phases by effort, with expected Spearman improvement of +0.04–0.07 and PR-AUC improvement of +0.05–0.10 without redesigning the core architecture.

---

## Gap Analysis

### Gap 1: RobustScaler Fitted on Tradeability Score (Data Leakage)

**Problem:** In `preprocess_ohlcv` (called at `train_lgbm.py` line 344), the `target_col` argument includes `'tradeability_score'`:
```python
"target_col": TARGET_COLS + ["tradeability_score"] + ...
```
This means the `RobustScaler` is `fit_transform`-ed on `df[scale_cols]` which includes **all numeric columns except those in the exclusion set**. The tradeability score column is NOT in the exclusion set at `preprocess.py` line 129 (`exclude = set(['Time', 'Open', 'High', 'Low', 'Close', 'Volume', 'Pivot', 'target', 'vol', 'outcomes', 'C_rel'])`). It's not excluded because `preprocess_ohlcv` only adds target_col values to `label_exclude` for the purpose of removing them from feature groups, but the scaler's `fit` call at line 164 (`scaler.fit_transform(df[scale_cols].values)`) uses the raw dataframe columns BEFORE filtering—and the tradeability_score is a numeric float column that passes through because exclusion lists are applied to feature group membership, not to the scaler's input.

**Evidence:** `train_lgbm.py` line 326-334 passes `target_col=TARGET_COLS + ["tradeability_score"] + ...` as part of the `preprocess_ohlcv_args`. The scaler at line 344 receives all numeric columns from `df_train` that aren't in the hard-coded exclude set. The `tradeability_score` column is a forward-looking derived column (generated by `generate_tradeability_scores` with a purge-gap classifier) and its values influence the scaling center/spread of genuine features.

**Impact:** Modest but real. The tradeability score has range [0.04, 1.0] with mean around 0.97 (most bars are tradeable). Including it shifts the RobustScaler's median/IQR for all genuine features. This is a form of mild target leakage.

**Proposed Fix:** Add `'tradeability_score'` to the hard-coded `exclude` set in `preprocess.py` line 129, AND ensure `target_col` entries that are intended as labels/weights (not features) are always added to `label_exclude` before the scaler fit. Specifically, modify `preprocess_ohlcv` so that **all** entries in `target_col` are excluded from the scaler's input, not just excluded from the final feature groups.

### Gap 2: Raw ATR Column Co-Scaled with Normalized Features (Scale Homogeneity)

**Problem:** The `atr` column computed at `train_lgbm.py` line 235 (`df["atr"] = ATR(...)`) is a raw dollar-denominated value (e.g., $1.50 for XAUUSD on M1). This column gets RobustScaled alongside features that are already z-scored (`fl_z_64`), ratio-normalized (`fl_atr14_pct` which is ATR/Close), or bounded [0,1] (`fl_stoch_k`). The result is a feature matrix where one column has vastly different semantics from others—a scaling mismatch that confuses tree-based models (which are scale-invariant but sensitive to relative split thresholds across features).

**Evidence:** In `preprocess_ohlcv`, the `atr` column is NOT in the `exclude` set, NOT a one-hot prefix, NOT a binary-like feature, and NOT in `normalized_passthrough`. So it gets RobustScaled at line 164. Meanwhile, `fl_atr14_pct` (atr/close) is already a ratio and gets scaled again. The model sees both the raw `atr` and the already-normalized variant.

**Impact:** LightGBM is partly scale-invariant but the split gain computation is influenced by feature distributions. Having a column with values ~1.5 alongside columns with values ~0.01 creates unnecessary tree depth consumption on a feature that adds no new information beyond `fl_atr14_pct`.

**Proposed Fix:** Either (a) remove raw `atr` from the feature matrix entirely (the `fl_atr14_pct` and `fl_atr28_pct` features already capture ATR-scaled information), or (b) add it to the exclusion set in `preprocess_ohlcv` at line 129. The raw `atr` is only needed for label computation (`calculate_trade_outcomes_capped`), not for prediction.

### Gap 3: Dropna Before Feature/Label Alignment in preprocess_ohlcv

**Problem:** `preprocess_ohlcv` calls `df = df.dropna().reset_index(drop=True)` at line 42, **before** it extracts outcomes, adjusts prices, and shifts targets. The initial `dropna` may discard rows that become invalid after the shift, but more critically, if `outcomes_col` values (`buy_class`, `sell_class`) are extracted at line 46 before the `dropna`, and then the `dropna` removes different rows, the alignment between `X`, `y`, and `outcomes` breaks. The current code works because `outcomes_col` is re-read from `df` after `dropna` at line 210, but this is fragile—it depends on `outcomes_col` being present in `df` columns after `dropna`.

**Evidence:** At `train_lgbm.py` line 329-333, `outcomes_col=['buy_class', 'sell_class']` is passed to keep these columns out of `X_train` but available in `proc_df_train`. The `preprocess_ohlcv` function extracts `outcomes` at line 46 (before `dropna`), then drops NaN at line 42, then re-extracts aligned outcomes at line 211. If the re-extraction at line 211 is 1-indexed wrong, the resolved-only masks at line 416-417 would be shifted.

**Impact:** Low probability but high severity—a 1-row misalignment would silently assign wrong class labels to every subsequent row's resolved mask, catastrophically corrupting the two-head regressor training.

**Proposed Fix:** Move the initial `dropna` to AFTER the outcomes extraction and alignment block. The outcomes should be extracted exactly once, and the `dropna` index should be applied uniformly to `X`, `y`, and outcomes. Better yet, extract outcomes from the same `df` that produced `X` and `y` (which already happens via `proc_df_train`), and add an explicit assertion that `len(X) == len(y) == len(outcomes)`.

### Gap 4: Walk-Forward Tuning Ignores Sign Accuracy

**Problem:** The model tuning metric is Spearman rank correlation (`tune_lgbm_by_spearman` at `train_utils.py` line 97-161), which only measures whether the model ranks bars correctly by magnitude. A model that predicts `+0.01` for all buy-winner bars and `-0.01` for all sell-winner bars gets a high Spearman if the ranks are correct, even if the **sign** (direction) is completely wrong. But the trading signal at `3.1 Backtest` cell 12 uses `expected_long - expected_short` which depends on **sign accuracy** of the individual long/short predictions.

**Evidence:** The threshold sweep in `train_lgbm.py` lines 829-868 computes sign hit rate **only** on `signed_quality` predictions, and **only** as a post-hoc analysis—never as part of the tuning loop. The tuning selects hyperparameters that maximize Spearman, which may actually hurt sign accuracy if the model learns to predict small-magnitude same-sign values for all bars.

**Impact:** The model with the best Spearman may not be the model with the best trading performance. This explains part of the gap between CV Spearman (~0.30) and actual backtest profitability.

**Proposed Fix:** Add a `sign_hit_rate` metric to the tuning loop in `tune_lgbm_by_spearman`. Compute it per fold as `np.mean(np.sign(y_true[va_idx]) == np.sign(pred))` for non-zero `y_true`. Add a composite score like `robust_score = 0.7*spearman_mean + 0.3*sign_hit_mean - penalty*std`. Alternatively, add a separate `tune_lgbm_by_sign_accuracy` function that uses the same fold structure but optimizes a threshold-dependent directional accuracy metric.

### Gap 5: Expected Value Formula Has Scale Mismatch

**Problem:** In the backtest signal generation (`3.1` cell 12):
```python
expected_long = pred_long * quality_long - p_sl_long
expected_short = pred_short * quality_short - p_sl_short
```
Here `pred_long = P(TP|long)` is a probability in [0, 1], `quality_long` is a `log1p(MFE/MAE)` value clipped to max 3.75 (mean ~0.66, median ~0.47), and `p_sl_long = P(SL|long)` multiplied by 1.0. The `quality_long` term dominates: a typical value is 0.30 * 0.47 = 0.14 vs 0.56 * 1.0 = 0.56 for the SL term. The SL penalty always outweighs the TP reward, making `expected_long` negative for most bars, which makes `pred_signed = expected_long - expected_short` essentially random.

**Evidence:** The `quality_long` distribution has mean 0.658 and P99 of 3.36 (from notebook output). P(TP|long) ≈ 0.37 (from class balance). So the expected TP component is ~0.37 * 0.658 ≈ 0.24. P(SL|long) ≈ 0.56 with penalty 1.0 = 0.56. Expected value ≈ 0.24 - 0.56 = -0.32 (negative for most bars on the long side).

**Impact:** The `pred_signed` signal is dominated by the SL probability, not by the quality prediction. The regression model's skill at predicting quality is essentially wasted in the trading formula because the SL penalty always swamps it.

**Proposed Fix:** Scale the quality prediction to be commensurate with the probability terms. Since the SL outcome is -1R (risk unit) and the TP outcome gives `R:R = tp_mult/sl_mult` R-multiples, the expected value should be:
```python
expected_long = (P(TP|long) * (TP_MULT/SL_MULT) - P(SL|long)) * quality_long
```
where `quality_long` serves as a confidence multiplier rather than being multiplied directly by probability. Alternatively, normalize `quality_long` to [0, 1] via `quality_long / TARGET_CLIP_MAX` so it acts as a scaling factor on the net probability. A cleaner formulation: use the product `P(TP|long) * quality_long` as an "expected quality" and compare it to `P(SL|long) * baseline_loss`, where `baseline_loss` is set to the median `quality_long` value (~0.47).

### Gap 6: 21% of Bars Have Zero Quality, Degrading Regression Training

**Problem:** From notebook output: `Zero share | long=0.209 short=0.215 signed=0.004`. This means 21% of bars have `long_quality <= 1e-12` (effectively zero), meaning both MFE and MAE are near zero—these are bars where price barely moves. These bars are included in the regression training with target=0.0, forcing the model to learn a flat prediction for noise bars, which consumes model capacity and dilutes the signal from informative bars.

**Evidence:** The `long_quality` and `short_quality` formulas at `train_lgbm.py` lines 242-243 use `np.log1p(outcomes["buy_MFE"] / (outcomes["buy_MAE"] + eps))`. When both MFE and MAE are tiny (chop bars), the ratio is ~1.0, `log1p(1.0) = 0.693`, but then clipping to 0 at the low end... wait, actually the clip is `np.clip(long_q, 0.0, TARGET_CLIP_MAX)` which floors at 0.0. The `log1p(MFE/MAE)` formula produces 0 only when MFE ≈ 0 and MAE > 0. These are bars where the price moved against the trade immediately and never recovered—genuine losing bars, and including them is correct. HOWEVER, the 21% figure is `(df['long_quality'] <= 1e-12).mean()`, which means 21% of bars have zero long quality after clipping. Some of these are genuinely bad trade entries (MAE hit immediately), but a significant fraction may be unresolved bars where both MFE and MAE are 0 because the trade timed out.

**Re-analysis:** After clipping, a bar with `MAE = 0` (price never went against) and `MFE > 0` gets `log1p(MFE/eps) = log1p(large) ≈ quality capped at 3.75`. A bar with `MFE = 0` and `MAE > 0` gets `log1p(0/MAE) = log1p(0) = 0.0 → clipped to 0.0`. A bar with both 0 (unresolved timeout) gets `log1p(0/eps) ≈ log1p(0) = 0.0`. So the 21% includes both true losing trades AND unresolved timeouts. The unresolved bars could have been resolved if given more time—labeling them as zero quality is a labeling error.

**Impact:** Approximately 7.7% of bars are unresolved timeouts (from class balance: `buy TP/SL/timeout = {0: 0.556, 1: 0.367, 2: 0.077}`). These bars get `quality=0` despite having no meaningful MFE/MAE data. Training the regression head on them (as zero) is noise.

**Proposed Fix:** In the two-head design, the regressor already trains only on resolved rows (`buy_class != 2`). Verify this is correctly excluding the 7.7% timeout bars. But even within resolved rows, SL-hit bars have `quality=0.0` because MAE > 0 and MFE = 0, which is correct labeling. The issue is that 21% target-zero bars create a mode at 0 in the target distribution that the model learns to predict as a default. **Better fix:** Use sample weighting to down-weight zero-quality bars (e.g., `weight = max(tradeability_score, 0.1)` so that flat bars get 10% the weight of high-quality bars). The `tradeability_score` is already available as a sample weight at `train_lgbm.py` line 373.

### Gap 7: MTF Feature Shift Consistency Between Training and Backtest

**Problem:** `add_multitimeframe_features` at `features.py` line 307 shifts HTF features by 1 HTF bar when `causal=True`. This shift is done **before** forward-filling to the base timeframe. In training, the base timeframe bars always align with completed HTF bars because the resample uses `closed='right'` semantics. In the backtest, the eval window may start mid-HTF-bar, causing the forward-fill to use a different HTF value than during training for the first few bars of the eval window.

**Evidence:** At `features.py` lines 304-308:
```python
if causal:
    df_tf_aligned = df_tf.copy()
    df_tf_aligned[mtf_cols] = df_tf_aligned[mtf_cols].shift(1)
```
Then at line 313: `df[col] = df_tf_aligned[col].reindex(df.index, method='ffill')`. The reindex with `method='ffill'` backfills from the last available HTF value. In training, this works because data starts from the beginning where HTF bars are well-defined. In backtest with `N_ROWS=10_000` (the eval window), the first few base timeframe bars may forward-fill from a different HTF context than during training.

**Impact:** The first `max(HTF_period_in_base_bars)` bars of the eval window have subtly different features than the corresponding training period, degrading holdout performance at the exact point where the backtest needs to generate signals.

**Proposed Fix:** In the backtest notebook (`3.1`), load at least `N_ROWS + max_base_bars_in_largest_HTF` extra bars for warmup, compute features on all of them, then discard the warmup before inference. Currently the backtest does `N_ROWS=10_000, WARMUP_BARS=7_000` but the warmup is only used for signal generation (rolling percentiles), not for feature computation—features are computed on exactly `N_ROWS` rows. Change to: load `N_ROWS + WARMUP_FEATURE_BARS` rows, compute features, then discard the first `WARMUP_FEATURE_BARS` rows. Set `WARMUP_FEATURE_BARS = 300` (the longest rolling window in the feature library: `fl_z_256` needs 256 bars).

### Gap 8: Time Features Dominate Importance, Risking Regime Overfitting

**Problem:** Hour sin/cos and DoW sin/cos features (`fl_hour_sin`, `fl_hour_cos`, `fl_dow_sin`, `fl_dow_cos`) plus session flags consistently rank as top-10 features by mutual information with `signed_quality`. This is pattern-recognized by many practitioners as a potential source of overfitting: the model learns "buy at 8am UTC, sell at 4pm UTC" rather than learning price-action patterns. When the market regime shifts (e.g., volatility clustering moves from London to NY session), the model's predictions degrade.

**Evidence:** In `add_feature_library` at `features.py` lines 1187-1199, time features use hard-coded UTC hours and fixed session boundaries (`tokyo: 0-9`, `london: 7-16`, `ny: 12-21`). These are NOT adjusted for DST, broker timezone, or instrument-specific active hours. The model trained on XAUUSD (which trades 23/5) will learn session patterns specific to gold—these patterns will fail on EURUSD or other instruments.

**Impact:** The feature selection pipeline (`select_features_pipeline` at `features.py` line 1386) almost certainly keeps these time features in the final 80 because they rank highly on mutual information during the in-sample selection period. This makes the model fragile to timezone shifts and session pattern changes.

**Proposed Fix:** (a) Add a `--drop-time-features` flag to `train_lgbm.py` that excludes `fl_hour_*`, `fl_dow_*`, `fl_session_*`, `hour`, `dayofweek` columns from the feature selection pool. (b) Run an ablation study: train two models (with and without time features), compare holdout Spearman. If time features account for >5% of Spearman, it confirms overfitting. (c) Replace absolute time features with **relative time features**: bars since session open, bars until session close, fraction of session elapsed. These are timezone-invariant.

### Gap 9: Feature Selection Uses Only Tail of Dataset

**Problem:** `select_features_pipeline` at `features.py` line 1386 is called with `fs_subset.tail(min(500_000, len(df_for_selection)))` at `train_lgbm.py` line 198. The mutual information computation at `features.py` line 1348 uses `mutual_info_regression` on only the most recent 500k bars. Features that are predictive across the full 1M-bar history but less so in the recent tail are dropped.

**Evidence:** `train_lgbm.py` lines 197-206:
```python
fs_subset = df_for_selection.tail(min(500_000, len(df_for_selection)))
selected_features = select_features_pipeline(
    fs_subset,
    target_col='signed_quality',
    n_features=SELECTED_FEATURE_COUNT,
    correlation_threshold=0.95,
    ...
)
```
The tail selection is for speed, but 50% of the dataset is discarded during feature selection. The remaining 500k bars cover the most recent ~2.4 years (for M1 data), which may be a single market regime.

**Impact:** Features that are predictive during earlier regimes (e.g., high-volatility 2020-2022) are discarded because they don't rank highly on 2024-2026 data. This reduces model robustness to regime changes.

**Proposed Fix:** Stratify the feature selection: split the 1M rows into 4 quarters, compute MI on each quarter independently, then take features that rank in the top 80 in ≥3 of 4 quarters. This costs 4x the MI computation time but produces a regime-robust feature set. Alternatively, use the full 1M rows with a random 200k-row sample per MI computation and average the MI scores over 5 bootstrap iterations.

### Gap 10: 3-Class Classifier Calibration and the Timeout Class

**Problem:** The 3-class classifier (`tune_lgbm_classifier`) optimizes for TP PR-AUC (`average_precision_score` on class 1) as the tuning metric (`train_utils.py` line 209). But the classifier is trained with `objective='multiclass'` on 3 classes (0=SL, 1=TP, 2=timeout). The tuning metric only evaluates how well the model separates TP from non-TP, ignoring whether timeout bars are being confused with SL bars. At inference (`3.1 Backtest` cell 12), only `proba[:, 1]` (P(TP)) and `proba[:, 0]` (P(SL)) are used. The timeout probability `proba[:, 2]` is computed but never used.

**Evidence:** `tune_lgbm_classifier` at `train_utils.py` lines 207-209:
```python
y_true_tp = (y[va_idx] == 1).astype(int)
tp_pr_auc = float(average_precision_score(y_true_tp, proba[:, 1]))
```
At inference in `3.1` cell 12:
```python
p_sl, p_tp = proba[:, 0], proba[:, 1]
# proba[:, 2] is silently dropped
```

**Impact:** The 3-class classifier is effectively used as a 2-class classifier at inference. The third class (timeout) only serves as a regularizer during training. But because timeouts are 7.7% of the data and have similar feature profiles to SL bars (both are "non-winning" outcomes), the classifier may learn to conflate SL and timeout distributions, reducing the discriminability of P(TP). A dedicated binary classifier (TP vs. rest) might outperform the multiclass approach.

**Proposed Fix:** (a) Use `objective='binary'` for the classifier, with the positive class being TP=1 and negative being all others (SL ∪ timeout = 0). This directly optimizes the probability used at inference. (b) Compute `P(SL) = 1 - P(TP) - P(timeout)` in the inference formula, so timeout probability explicitly reduces the sum and acts as a "do not trade" gate. (c) Add a `P(timeout)` gate to the signal generation: if `proba[:, 2] > 0.5`, suppress the trade signal regardless of other probabilities (the bar is likely to not resolve within horizon).

### Gap 11: MLP Two-Head Architecture Under-Tuned

**Problem:** The `TwoHeadMLP` at `Util.py` line 48 uses a fixed architecture: shared trunk of (128, 64) with ReLU activations, no batch normalization, no dropout. The training loop at `train_two_head_mlp` uses fixed hyperparameters (lr=1e-3, epochs=60, batch_size=4096) with no learning rate schedule or early stopping. The combined loss is `CE + MSE` with equal weight—no tuning of the loss balance.

**Evidence:** From `train_lgbm.py` lines 76-78:
```python
DL_HIDDEN_SIZES = (128, 64)
DL_EPOCHS = 60
DL_LR = 1e-3
DL_BATCH_SIZE = 4096
```
No validation set is used during MLP training (only the training set). The MLP never sees a holdout set during training, so there's no early stopping—it always trains exactly 60 epochs regardless of overfitting.

**Impact:** The MLP achieves Spearman 0.27 vs LGBM 0.31 (a 13% gap). This gap is partly architectural: the MLP has ~(80 × 128 + 128 × 64 + 64 × 3 + 64 × 1) = 18,752 parameters, which is tiny compared to the 5,000-tree LightGBM. Adding batch normalization, dropout, and a learning rate scheduler could close the gap significantly.

**Proposed Fix:** (a) Add `nn.BatchNorm1d` between linear layers in the trunk. (b) Add dropout (0.2) after each activation. (c) Use `torch.optim.lr_scheduler.ReduceLROnPlateau` with patience=10. (d) Split 10% of the training set for validation and use early stopping. (e) Add a tunable loss weight parameter `alpha` for `loss = CE_loss + alpha * MSE_loss` and grid search `alpha ∈ {0.1, 0.5, 1.0, 2.0, 5.0}`.
